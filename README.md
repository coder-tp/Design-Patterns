欢迎来到设计模式，有人已经解决了你的问题。
我们的目的是：**如何“优雅地”应用设计模式解决实际、大规模的问题。**
# 一、学习模式关键路径图：
理解痛点 -> 学习原则 -> 探索模式 -> 书写实例代码 -> 重构应用 -> 分析源码 -> 融入项目
# 二、注重理解“why” 而不是“how”
>接下来的23天，我们重点针对不同的设计模式按照这个思路去展开学习。

**1.场景**
**​2.它解决了什么问题？​**​ 什么场景下会产生这个痛点？
​**​3.为什么常用解决方案不好？​**​ 会带来什么耦合、僵化或难以复用？
**​4.模式的方案如何解决这些问题？​**​ 它是如何松耦合、灵活扩展、支持复用的？
**​5.它的优缺点和适用场景是什么？​**​ 没有万能模式，明确边界更重要。
​**​6.模式之间的关系？​**​ （例如，工厂方法常用来创建策略模式需要的策略对象）。
# 三、小练习

> 做练习。自己做笔记。


# 四、寻找“坏味道”
>在自己的旧项目或GitHub上找一些合适的开源小项目（功能清晰但设计一般的那种）。特别关注：条件语句过多处理不同行为（适合策略）、直接新建对象/初始化复杂（适合工厂/建造者）、类依赖过紧（适合适配器/门面）。

>当你纠结代码怎么写、发现修改一处要动十处、感觉代码难以测试时，暂停思考：“这痛点的本质是什么？哪个设计模式能解决？”

# 五、模式驱动重构
>请有意识地思考用哪个模式能优雅解决这些痛点。安全操作：分支重构、写测试、小步前进。

# 六、学习参考资源：
>找了一些比较经典的资源进行展开学习

 [1]: 《Head First 设计模式》​强烈推荐作为入门首选！​语言风趣幽默，图文并茂，用大量现实生活中的比喻来解释抽象概念，非常容易上手。案例也比较实用。先理解思想再学代码。  
 [2]: 《设计模式：可复用面向对象软件的基础》(GoF - Gang of Four)：​​权威经典，必读！这是设计模式的起源。在《Head First》打底后，深入阅读GoF对23种模式的原始定义、动机、结构和效果。理解其严谨性和抽象层次。适合精读和反复查阅。  
 [3]: 《深入设计模式》： 配套有在线互动图表（refactoring.guru）。解释清晰，有大量UML图、代码示例和现实世界类比，可以作为《Head First》或GoF的补充或替代。  
 [4]: 《测试驱动开发》:TDD会迫使你设计低耦合模块（方便测试），而这恰恰是许多模式（工厂、策略、命令）带来的收益。模式让TDD更轻松！在编写测试时感受模式如何改善可测性。  
 [5]: 菜鸟教程：设计模式系统学 https://www.runoob.com
